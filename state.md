## 🆕 CAPI for CRM (7/14 구축) — 가장 중요한 변화

**문제였던 것: 메타는 "폼 작성자"만 알았지 "구매자"를 몰랐다.**

지금까지 메타에 간 신호는 `Lead`(폼 제출) 하나뿐. 그래서 알고리즘은 **폼을 잘 내는 사람**을 찾아왔다.
25-34는 폼을 잘 낸다. **오더를 안 할 뿐이다.** 6월 MLST(25-34가 64~78% · 오더 0)와
7월 채리 붕괴가 **같은 병**인 이유.

**해결: 노션 상태 변화를 메타에 다시 쏜다.**

```
학부모 폼 제출 → 메타가 leadgen_id 발급 → 노션에 '리드ID' 저장
     ↓
상담팀이 상태를 '방문'/'오더'로 변경
     ↓
updateLeadFields → sendFunnelCAPI → 메타 CAPI POST
     ↓
메타: "이 리드가 오더했구나" → 오더하는 사람을 닮은 사람을 찾음
```

**구현 내역 (아라위키 자동화 Code.gs)**

| 항목 | 내용 |
|---|---|
| 노션 신규 속성 | **`리드ID`** (텍스트) |
| 신규 함수 | `sendFunnelCAPI` / `extractCapiFieldsFromPage` / `capiSha256` / `capiNormalizePhone` / `capiStripPrefix` |
| 수정 함수 | `sendMetaCAPI` (lead_id를 **custom_data → user_data**로 이동) / `sendBulkToNotion` (리드ID 저장) / `updateLeadFields` (**CAPI 발송 추가**) |
| 1회성 | `backfillLeadIds` (phone 매칭으로 리드ID 소급) / `sendRecentFunnelEvents` (최근 7일 방문·오더 전송) |
| 테스트 | `testCapiConnection` → **200 / events_received:1 확인 완료** |

**이벤트 매핑**

| 노션 상태 | 메타 이벤트 | 매출값 |
|---|---|---|
| 방문 | `Schedule` | — |
| 오더 · 미납 | `Purchase` | 노션 raw × 1,000 |
| 결보 · 취소 · 당취 | **전송 안 함** (매출 0원) | — |

⚠️ **CAPI 제약 (반드시 기억)**
- **`event_time`은 7일 이내만 허용.** 그 밖은 메타가 **거부**한다 → **과거 오더 소급 전송 불가**
- `lead_id`는 **`user_data`에** 넣어야 매칭됨. `custom_data`에 있으면 그냥 메모
- `action_source`는 **`system_generated`** (CRM 이벤트)
- 효과 발현까지 **약 1개월** (메타 공식). 8월 중순 예상

**다음 단계 (3~4주 후 판단)**
- Meta Leads Funnel 구성 (폼제출 → 방문 → 오더 순서 정의)
- 성능목표를 **"전환 리드 극대화"**로 전환
  - 조건: 최적화 이벤트가 **주 50건** 필요 → **'방문'(전환율 11.1%, 주 52건)**이 후보. **오더는 주 13건이라 불가**
  - ⚠️ 50건은 **광고세트 단위**. 활성 4개로 나누면 각 13건 → **광고세트 통합 필요할 수 있음** (별도 결정)

## 측정 원칙 (7/11 확립 — 가장 중요)

**관통하는 문제였던 것: 분모와 분자가 서로 다른 것을 세고 있었다.**

**채널 성격이 다르면 측정 방식도 달라야 한다**
- **유량**(메타·카카오) = 월별
- **저량**(텐핑) = 코호트 (잔액 0 = 코호트가 닫힘)
- **정액**(브랜드검색) = **일할** (월중엔 경과일 비율)

**ROAS는 두 기준을 병기한다 — 격차 자체가 진단이다**
- **코호트 ROAS** = 그달 광고비 ÷ 그달 유입 리드의 누적 매출 → **채널·소재 생사 판단**
- **결제일 ROAS** = 그달 광고비 ÷ 그달 결제된 매출 → **경영 보고**
- 격차 없음 = 전환 시차 짧은 건강한 채널 / 격차 큼 = 이번 달 매출은 과거 리드가 만든 것

**교정된 수치 (전부 착시였음)**

| | 착시 | 실제 |
|---|---|---|
| 네이버 | 64.4 | **2.22** (분모 GFA만 / 분자 브검 매출) |
| 카카오 | 5.65 | **0.88** (코호트 · 본전 미달) |
| 텐핑 | 1.10 | **2.64** (누적 코호트) |
| 메타 | 12.49 | **6.1** (분모가 5일치, 분자가 11일치) |

⚠️ **월중 진행 상황을 볼 땐 항상 분모 기간을 분자와 맞출 것.** 7/13에도 리포트에서 이 실수를 재현했다(7월 브검 매출 13일치를 광고비 31일치로 나눔 → ROAS 11.55, 실제 일할 27.26).

## 소재 판정 원칙 (6월 결산으로 실증)

**CPL 단독 판단 금지. ROAS로 판단한다.**

**CPM은 오디언스 품질의 진단 신호다**
- **CPM 1만 미만 5종 → 오더 0. 예외 없음**

- 인과: 싼 CPM = 싼 오디언스(학생) = 싼 리드 = 안 팔림. **CPL이 낮았던 이유가 곧 오더 0이었던 이유**
- **7/14 노트공부법 OFF도 이 원칙 적용** (CPM 8,550 → CPL 67,672)

## 주의사항 (다음 세션 Claude가 반드시 알 것)

**측정**
- **소재 생사 판단은 ROAS 기준. CPL 단독 판단 금지.** 여러 지표(CPL·ROAS·CPM·완주율·빈도·연령)를 복합적으로 보고, **최종 판단은 상준씨 몫**
- 월중 지표를 볼 땐 **분모 기간을 분자와 맞출 것** (브검 일할 등)
- 주간 ROAS는 시차 때문에 착시 — 운영 지표(CPL·빈도·소재 반응)만 볼 것
- **하루치로 단정하지 말 것.** 7/14에 채리 3일 붕괴를 "확정적"이라 했으나 당일 반등함

**함정 (전부 실제로 밟은 것)**
- **프로젝트가 2개다.** getLeadStats는 "대쉬보드", 카카오·네이버 수집과 CAPI는 "아라위키 자동화". 붙여넣기 전 프로젝트명 확인
- `openById`로 연 다른 스프레드시트의 Date는 **`instanceof Date`가 false로 떨어짐** → `typeof v.getTime === 'function'`으로 판별
- 카카오 SignUp은 **`conv_signup_1d`만**. 정규식으로 `sign_up` 포함 키를 재합산하면 `cost_per_conv_signup`이 섞여 오염됨(181693.398 사건)
- `clearContents()`는 서식을 안 지움 → 옛 % 서식 잔존 시 118이 11800%로 표시. `setNumberFormat('0')` 범위에 **합계행 포함** 필수
- 반올림은 **raw를 다 더한 뒤 마지막에 한 번만**. 개별 반올림값을 되더하면 오차 누적(8422 vs 8421)
- 메타 데일리체크는 `results`만. **`lead` 필드 금지**
- 메타 예산 변경 시 **MCP가 자동으로 PAUSED를 걸 수 있음** → `ads_activate_entity`로 다시 켜야 함
- **`filtering`에 `campaign.id` IN 필터가 빈 결과를 뱉을 때가 있음** → `campaign.name` CONTAIN으로 우회
- **CAPI `event_time` 7일 제한.** 소급 전송 시도 금지
- **Advantage+ 최소 연령은 25가 천장.** 연령/성별/관심사는 "제안"일 뿐 강제 아님

**소스 체계**
- `naver_brand` = 브랜드검색 / `naver_gfa` = GFA / 파워컨텐츠는 DB 유입 없음
- 랜딩 URL은 브검 검수 재신청 이슈로 미변경 → **Apps Script가 저장 시점에 `naver` → `naver_brand` 변환**
